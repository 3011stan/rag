# Metadata

Metadata connects curated knowledge in StanOS/Obsidian with retrieval behavior in the RAG backend.

## Goals

- Preserve technical facts generated by loaders.
- Allow user-curated semantic and editorial metadata.
- Keep ingestion backward-compatible.
- Prepare the corpus for future retrieval preferences and soft boost ranking.

## Reserved Technical Fields

These fields belong to the system/loaders and must not be accepted from user-provided curation metadata:

- `filename`
- `content_type`
- `source_type`
- `pages`
- `checksum`

If user-provided metadata contains one of these fields, the request should be rejected.

## Curation Fields

These fields may be provided manually during ingestion:

- `type`
- `layer`
- `category`
- `platform`
- `source_kind`
- `source_quality`
- `evergreen`
- `visibility`
- `source_url`
- `author`
- `created`
- `captured_at`
- `tags`

`ingestion_status` is intentionally not persisted in the RAG backend. It is a StanOS/Obsidian workflow field used before ingestion.

## Precedence

Precedence rule:

```txt
loader wins technical/reserved fields;
curation wins semantic/editorial fields when provided;
curation is optional.
```

Examples:

- If the loader sets `filename`, user metadata cannot change it.
- If the loader extracts an empty or weak `author`, curated `author` may replace it.
- If no curation metadata is provided, loader metadata is used as-is.

## Backward Compatibility

Existing records may only have basic loader metadata.

Missing curation metadata means:

- No boost for that missing field.
- No ingestion failure.
- No retrieval failure.
- No exclusion from semantic search.

Existing documents should later receive manual metadata backfill because the current corpus is small.

## Example Request

```bash
curl -X POST http://localhost:8080/rag/ingest \
  -F 'file=@/path/to/notes.md;type=text/markdown' \
  -F 'metadata={"type":"knowledge_asset","layer":"foundations","category":"storytelling","platform":"general","source_kind":"article","source_quality":"high","visibility":"private","tags":["narrative","retention"]}'
```

## Future Retrieval Use

Metadata should be used as retrieval preferences, not hard filters, by default.

Future `/rag/ask` requests may send preferences such as:

```json
{
  "question": "How should I structure a technical reel?",
  "top_k": 5,
  "preferences": {
    "layers": ["platform_specific", "self_knowledge"],
    "categories": ["storytelling"],
    "platforms": ["reels"],
    "source_kinds": ["note", "article"],
    "source_quality": ["high"]
  }
}
```

The backend should prefer matching chunks without excluding semantically useful chunks from other layers.

The ask preferences contract accepts plural array fields only:

- `layers`
- `categories`
- `platforms`
- `source_kinds`
- `source_quality`

`visibility` is intentionally not an ask preference. It should remain a corpus/access decision rather than a ranking hint.

## Metadata in Answer Sources

`POST /rag/ask` may return metadata in each source item.

Only a sanitized allowlist is exposed publicly:

- `type`
- `layer`
- `category`
- `platform`
- `source_kind`
- `source_quality`
- `evergreen`
- `visibility`
- `source_url`
- `author`
- `created`
- `captured_at`
- `tags`
- `source_type`
- `content_type`
- `filename`
- `pages`

Fields outside this allowlist, such as `checksum`, must not be returned by the public ask response.
